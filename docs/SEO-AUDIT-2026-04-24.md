| [public/images/carport-residential-completed.jpg](public/images/carport-residential-completed.jpg) | 772 KB | site assets — verify usage |
| [public/images/porch-cover-lean-to.jpg](public/images/porch-cover-lean-to.jpg) | 764 KB | `/services` hub (mapped to `rv-covers`) |
| [public/images/red-iron-frame-hero.jpg](public/images/red-iron-frame-hero.jpg) | 704 KB | hero on homepage, services hub, thank-you, locations fallback (highest-traffic photo) |
| [public/images/carport-truck-concrete-hero.jpg](public/images/carport-truck-concrete-hero.jpg) | 696 KB | `/military` hero + `/services/turnkey-carports-with-concrete` |
| [public/images/double-carport-install.jpg](public/images/double-carport-install.jpg) | 656 KB | `/services` (mapped to `barns`) + `/services/barns` |

`next/image` is used everywhere these are referenced (verified via grep), so Next.js does on-demand WebP/AVIF transcoding + responsive sizes at request time. **But:** Next.js still has to fetch + decode the source PNG/JPEG once on first request, which:
- Hits Vercel's image-optimization function with a >700 KB source on every regional cache miss → slow LCP on first regional hit.
- Inflates `next build` time (pre-render fetches on prebuilt routes).

**Recommendation:** Run sources through `cwebp -q 82` (or similar) targeting ≤300 KB for the JPEGs and ≤200 KB for the logo PNG. The lion PNG specifically should drop to ≤100 KB. Drop into `public/images/`, leave Next.js to handle responsive variants.
